Introducción a las Bases de Datos NoSQL

Las bases de datos NoSQL ocupan en la actualidad una parcela importante en el desarrollo de aplicaciones, principalmente aquellas relacionadas con el Big Data.

¿Qué son las Bases de Datos NoSQL?

Las bases de datos NoSQL están optimizadas para aplicaciones que requieren grandes volúmenes de datos, baja latencia y modelos de datos flexibles, lo que se logra mediante la flexibilización de algunas de las restricciones de consistencia de datos o por la existencia de un esquema fijo para el modelo de datos.

Características Principales:

Este tipo de bases de datos se caracterizan porque no utilizan el esquema tabular de filas y columnas. Utilizan un modelo de almacenamiento optimizado para el tipo de datos que van a almacenar. En este sentido encontramos datos que se pueden almacenar como pares clave-valor simple, como documentos JSON o como un grafo que consta de aristas y vértices.

Flexibilidad

Generalmente ofrecen esquemas flexibles que facilitan un desarrollo iterativo

Escalabilidad

Están diseñadas para escalar usando clústeres distribuidos

Alto Rendimiento

Cada tipo está optimizada para trabajar con el tipo de datos que almacena

Altamente Funcional

Proporcionan APIs y tipos de datos diseñados específicamente para cada uso

Comparación con Bases de Datos Relacionales:
Característica Bases de Datos Relacionales Bases de Datos NoSQL
Modelo de Datos Tablas con filas y columnas Documentos, Grafos, Clave-Valor, Columnas
Esquema Fijo y rígido Flexible y dinámico
Escalabilidad Principalmente vertical Principalmente horizontal
Consistencia ACID fuerte Eventual o flexible
Consultas SQL estándar APIs específicas
Uso Ideal Datos estructurados y transaccionales Big Data, contenido no estructurado

Bases de Datos Orientadas a Documentos

El concepto central de una base de datos orientada a documentos es el documento. Cada implementación de base de datos utilizará tipos de documentos distintos, pero todas ellas coinciden en encapsular y codificar datos o información siguiendo algún formato estándar.

Formatos de Documentos:

Algunas de las codificaciones más utilizadas son:

  • JSON (JavaScript Object Notation): Formato ligero de intercambio de datos
  • BSON (Binary JSON): Representación binaria de JSON
  • XML (eXtensible Markup Language): Lenguaje de marcado extensible
  • YAML (YAML Ain't Markup Language): Formato de serialización de datos
Flexibilidad del Esquema:

Al igual que en los sistemas de bases de datos relacionales los documentos actúan como registros (o filas), sin embargo, la flexibilidad hace que dos registros no sean iguales. Esto hace que podamos encontrarnos con registros cuyos conjuntos de campos son distintos.

Ejemplo: Documentos con Estructura Diferente

// Documento 1: Información básica { "empleadoId": "CA1", "nombre": "Carlos García", "salario": 45000 } // Documento 2: Información extendida { "empleadoId": "MG1", "nombre": "María González", "salario": 50000, "proyectos": [ "gestión-riesgos", "detección-fraude" ], "departamento": { "nombre": "IT", "ubicacion": "Madrid" } }

Ambos serían documentos válidos. El primer documento solo tiene información básica de un empleado, en cambio el segundo documento tiene más información, añadiendo así más datos sobre el empleado.

Ventajas de las Bases de Datos Documentales:
  • Flexibilidad ilimitada: Permite evolucionar el esquema del documento en el tiempo
  • Modelo natural: Los documentos se mapean directamente a objetos en lenguajes de programación
  • Consultas anidadas: Permite consultar dentro de estructuras anidadas
  • Desarrollo ágil: Facilita el desarrollo iterativo sin migraciones de esquema
Casos de Uso:

Este tipo de base de datos es adecuado para:

  • Gestión de catálogos: Productos con atributos variables
  • Perfiles de usuario: Información personalizada por usuario
  • Sistemas de gestión de contenido: Donde cada documento es único y evoluciona con el tiempo
  • Aplicaciones web modernas: APIs REST que trabajan con JSON
Bases de Datos Documentales Principales:

MongoDB

Base de datos NoSQL orientada a documentos creada por la compañía 10gen en el año 2007. Se caracteriza por que almacena los datos en documentos de tipo JSON con un esquema dinámico denominado BSON.

// Ejemplo de consulta en MongoDB db.empleados.find({ "salario": { $gt: 50000 }, "departamento.nombre": "IT" })

Couchbase

Base de datos documental optimizada para aplicaciones interactivas. Combina la flexibilidad de JSON con la potencia de SQL para consultas.

// Ejemplo de consulta en Couchbase (N1QL) SELECT nombre, salario FROM empleados WHERE salario > 50000 AND departamento.nombre = "IT";

Bases de Datos Orientadas a Grafos

En las bases de datos orientadas a grafos se administran dos tipos de información: el nodo y la arista. Los nodos representan las entidades y las aristas representan las relaciones que las unen.

Componentes de un Grafo:
  • Nodos (Vértices): Representan las entidades (personas, lugares, cosas)
  • Aristas (Edges): Representan las relaciones que unen los nodos
  • Dirección: Las aristas pueden tener dirección que indicará la naturaleza de la relación
  • Propiedades: Tanto nodos como aristas pueden tener propiedades (atributos)
Características de las Aristas:

Las aristas siempre deben tener:

  • Un nodo de inicio
  • Un nodo de final
  • Un tipo (que define la naturaleza de la relación)
  • Una dirección (opcional, puede ser bidireccional)

No existe un límite para la cantidad ni el tipo de relaciones que un nodo puede tener.

Potencial de las Bases de Datos de Grafos:

El potencial de las bases de datos de grafos radica en las relaciones, debido a que gracias a ellas se puede navegar por los nodos de la base de datos de manera rápida, ya que las relaciones entre los nodos no se calculan en los tiempos de consulta, sino que se mantienen en la base de datos.

Casos de Uso:

Este tipo de bases de datos son muy utilizadas en:

  • Redes sociales: Para modelar relaciones entre usuarios
  • Sistemas de recomendación: Para encontrar productos o contenido relacionado
  • Detección de fraude: Para identificar patrones y conexiones sospechosas
  • Análisis de redes: Para estudiar conexiones y patrones complejos
  • Gestión del conocimiento: Para representar conocimiento interconectado

Ejemplo: Grafo para Red Social

Estructura del Grafo

Nodos:

  • Persona (Juan, María, Pedro)
  • Restaurante (El Buen Sabor, La Pizzería)
  • Ciudad (Madrid, Barcelona)

Relaciones:

  • amigos (Persona → Persona)
  • likes (Persona → Restaurante)
  • viveEn (Persona → Ciudad)
  • localizadoEn (Restaurante → Ciudad)

Consultas Típicas

// Encontrar todos los restaurantes en Madrid MATCH (r:Restaurante)-[:localizadoEn]->(c:Ciudad {nombre: "Madrid"}) RETURN r; // Encontrar amigos de amigos MATCH (p1:Persona)-[:amigos]->(p2:Persona)-[:amigos]->(p3:Persona) WHERE p1.nombre = "Juan" RETURN p3; // Restaurantes que les gustan a mis amigos MATCH (p1:Persona {nombre: "Juan"})-[:amigos]->(p2:Persona)-[:likes]->(r:Restaurante) RETURN DISTINCT r;
Propiedades y Etiquetas:

A los nodos se les pueden añadir etiquetas (propiedades), como nombre o dirección, que añadan más información sobre la entidad. De la misma manera se pueden añadir etiquetas a las relaciones, que aportarán más información sobre la relación, por ejemplo, valoración a la relación likes que indique el grado de satisfacción de una Persona con un Restaurante.

// Ejemplo: Nodo con propiedades { "id": "persona_1", "tipo": "Persona", "propiedades": { "nombre": "Juan Pérez", "edad": 30, "email": "juan@email.com" } } // Ejemplo: Arista con propiedades { "tipo": "likes", "desde": "persona_1", "hacia": "restaurante_1", "propiedades": { "valoracion": 5, "fecha": "2024-01-15", "comentario": "Excelente comida" } }
Bases de Datos de Grafos Principales:
  • Neo4j: La más utilizada, con lenguaje de consulta Cypher
  • OrientDB: Base de datos multi-modelo que soporta grafos
  • Infinite Graph: Base de datos de grafos distribuida
  • AllegroGraph: Base de datos de grafos RDF

Bases de Datos Clave-Valor

Una base de datos clave-valor se asemeja a una tabla hash grande. Cada valor de datos está asociado con una clave única y se usará esta clave para almacenar los datos mediante el uso de una función hash apropiada.

Modelo de Datos:

Al igual que las bases de datos orientadas a documentos, las bases de datos clave-valor no necesitan que el esquema esté definido previamente. El modelo es extremadamente simple:

  • Clave: Identificador único (string, número, etc.)
  • Valor: Datos asociados (puede ser cualquier tipo: string, número, objeto, etc.)

Ejemplo de Almacenamiento Clave-Valor

Estructura Simple

// Ejemplo de pares clave-valor "usuario:123" → { "nombre": "Juan Pérez", "email": "juan@email.com", "edad": 30 } "producto:456" → { "nombre": "Laptop", "precio": 999.99, "stock": 50 } "sesion:abc123" → { "usuario_id": 123, "login_time": "2024-01-15T10:30:00Z", "expires": "2024-01-15T11:30:00Z" }
Características:
  • Modelo más sencillo: El modelo de datos más simple de todos los tipos NoSQL
  • Alto rendimiento: Muy optimizado para búsquedas simples mediante el valor de la clave
  • Escalabilidad: Escalan fácilmente de forma horizontal
  • Sin esquema: No requiere definición previa de estructura
Limitaciones:

Los sistemas de bases de datos clave-valor están muy optimizados para aplicaciones que realizan búsquedas simples mediante el valor de la clave, sin embargo, son menos adecuados para aquellas aplicaciones que necesitan consultar datos en diferentes tablas de claves-valores o realizar operaciones complejas como JOINs.

Casos de Uso:

Este tipo de base de datos es adecuada para:

  • Mantenimiento de sesión: Almacenar sesiones de usuarios en aplicaciones web
  • Carro de compra: Almacenar el carro de la compra de un e-commerce
  • Cache: Sistema de caché de alto rendimiento
  • Configuración: Almacenar configuraciones de aplicaciones
  • Contadores: Contadores y métricas en tiempo real
  • Colas de mensajes: Sistemas de mensajería simple
Bases de Datos Clave-Valor Principales:

Apache Cassandra

Modelo de base de datos NoSQL basado en el almacenamiento clave-valor y de código abierto escrita en Java. Cassandra opera de manera distribuida consiguiendo así escalabilidad lineal y alta disponibilidad.

Usuarios conocidos: Twitter, Wikimedia, Apple, Netflix

// Ejemplo de uso en Cassandra INSERT INTO usuarios (id, nombre, email) VALUES (123, 'Juan Pérez', 'juan@email.com'); SELECT * FROM usuarios WHERE id = 123;

Redis

Base de datos en memoria de tipo clave-valor, conocida por su alto rendimiento. Soporta múltiples estructuras de datos: strings, hashes, listas, conjuntos, etc.

// Ejemplo de uso en Redis SET usuario:123 "Juan Pérez" GET usuario:123 HSET usuario:123:detalles nombre "Juan" email "juan@email.com" HGETALL usuario:123:detalles

Amazon DynamoDB

Base de datos NoSQL gestionada de AWS que ofrece rendimiento de milisegundos de un solo dígito a cualquier escala. Soporta tanto documentos como clave-valor.

// Ejemplo de uso en DynamoDB { "TableName": "usuarios", "Item": { "id": {"N": "123"}, "nombre": {"S": "Juan Pérez"}, "email": {"S": "juan@email.com"} } }

Otras bases de datos clave-valor conocidas: Riak, Voldemort, Amazon Dynamo

Actividades de Refuerzo

Actividad 1: Trabajar con MongoDB

Duración: 30 minutos

Objetivo: Familiarizarse con bases de datos orientadas a documentos.

Tareas:

  1. Instala MongoDB o usa MongoDB Atlas (gratuito)
  2. Crea una base de datos para un sistema de blog
  3. Inserta documentos con estructura flexible (algunos con comentarios, otros sin)
  4. Realiza consultas básicas y agregaciones
  5. Compara con cómo harías lo mismo en una base de datos relacional
30 min

Actividad 2: Modelar una Red Social con Neo4j

Duración: 40 minutos

Objetivo: Entender las bases de datos orientadas a grafos.

Tareas:

  1. Instala Neo4j Desktop o usa Neo4j Aura (gratuito)
  2. Crea nodos para usuarios, posts y comentarios
  3. Establece relaciones: sigue, le_gusta, comenta
  4. Realiza consultas para encontrar amigos de amigos
  5. Encuentra usuarios que les gustan los mismos posts
40 min

Actividad 3: Sistema de Cache con Redis

Duración: 25 minutos

Objetivo: Aprender a usar bases de datos clave-valor.

Tareas:

  1. Instala Redis o usa Redis Cloud (gratuito)
  2. Crea un sistema de cache para datos de usuarios
  3. Implementa un carro de compra usando estructuras de datos de Redis
  4. Usa expiración automática para sesiones
  5. Crea un ranking usando conjuntos ordenados
25 min
